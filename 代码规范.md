# 当前工程代码规范

本文适用于当前 `Hero` 工程的手写业务代码，范围包括 `User/APP`、`User/APP_Support`、`User/BSP`、`User/Devices`、`User/Algorithm`。CubeMX 管理项按 CubeMX 流程修改。

## 模块归属

- `User/APP/*_task.c` 负责 RTOS 任务入口、全局控制对象、模块初始化调用、周期调度和弱接口声明。
- `User/APP/*_task.h` 负责任务数据结构、模块接口声明和跨模块状态字段。
- `User/APP_Support/common/robot_param.h` 负责机器人级宏、CAN ID、通道映射、底盘/云台/发射/功率等跨模块参数。
- `User/APP_Support/chassis/chassis_behaviour.*` 负责遥控器、键鼠、掉线状态到底盘行为模式和底盘控制模式的映射。
- `User/APP_Support/chassis/chassis_calculate.*` 负责底盘运动学正解、逆解、坐标旋转和运动学限幅。
- `User/APP_Support/chassis/Omni_chassis.*` 负责底盘初始化、反馈更新、目标生成、速度规划、轮速控制、制动补偿和 CAN 发送。
- `User/APP_Support/power_control/*` 负责底盘功率预测、功率限幅、电流缩放和功率模块通信。
- `User/APP_Support/gimbal/gimbal_behaviour.*` 负责云台行为模式映射。
- `User/APP_Support/gimbal/yaw_pitch_direct.*` 负责 yaw-pitch 云台反馈、目标生成、PID/前馈/摩擦补偿和 MIT/3508 输出。
- `User/APP_Support/gimbal/target_curve.*` 负责云台目标曲线和轨迹约束。
- `User/APP_Support/shoot/shoot_task.*` 负责发射弱接口封装，`shoot_3508.*` 负责 3508 发射机构控制。
- `User/BSP/*` 负责板级外设封装、CAN/UART/TIM/DWT 和遥控器底层数据。
- `User/Devices/*` 负责设备级封装和调试输出，VOFA 固定按 6 通道发送。

新增功能先判断职责归属，沿已有模块文件、结构体、宏分组和调用链路接入。

## 控制链路

底盘任务链路固定为：`chassis_init` -> `chassis_set_mode` -> `chassis_mode_change_control_transit` -> `chassis_feedback_update` -> `chassis_set_contorl` -> `chassis_control_loop` -> `chassis_send_cmd` -> `osDelay(CHASSIS_CONTROL_TIME_MS)`。

底盘控制循环固定为：速度规划 -> `chassis_body_feedforward_update` -> `chassis_body_velocity_brake_update` -> `chas_inv_cal` -> `PID_Calc_Jump` -> `chassis_dynamic_current_limit_update` -> `chassis_power_control` -> CAN 电流发送。

云台任务链路固定为：`gimbal_init` -> `shoot_init` -> `gimbal_set_mode` -> `gimbal_mode_change_control_transit` -> `gimbal_feedback_update` -> `gimbal_set_control` -> `gimbal_control_loop` -> `gravity_comp_execute` -> `shoot_control_loop` -> `gimbal_send_cmd` -> `vTaskDelayUntil(GIMBAL_CONTROL_TIME)`。

任务文件提供 `__weak` 接口，具体模块用同名函数实现；受 `ROBOT_CHASSIS`、`ROBOT_GIMBAL`、`ROBOT_FRICTION` 等宏选择的实现函数保留 `__attribute__((used))`。

## 参数和配置

- 机器人级和跨模块参数集中放在 `User/APP_Support/common/robot_param.h`。
- 底盘参数按几何映射、任务输入、速度规划、量纲换算、轮速 PID、电流限幅、yaw 控制、整车前馈制动、单轮前馈制动分组。
- 云台参数按电机 ID、角度限位、遥控灵敏度、PID、前馈、摩擦补偿、MIT 协议范围分组。
- 发射参数优先放入 `User/APP_Support/shoot/shoot_task.h` 或 `shoot_3508.h` 的既有参数区。
- 通信和裁判参数优先放入对应通信/裁判模块头文件，跨模块公共量再进入 `robot_param.h`。
- 数值宏名带单位语义，角度默认 `rad`，角速度默认 `rad/s`，平移速度默认 `m/s`，电流命令默认 3508 电流命令计数。
- CubeMX 管理项修改流程固定为：先说明 CubeMX 路径、建议值、生成代码后的影响文件，再由操作者确认或在 CubeMX 中修改。

## 命名规范

- 类型命名使用小写模块名加语义后缀：结构体用 `_t`，枚举用 `_e`，示例为 `chassis_move_t`、`gimbal_control_t`、`chassis_mode_e`。
- 控制函数使用模块前缀加动作：`chassis_*`、`gimbal_*`、`shoot_*`、`VOFA_*`。
- 全局控制对象使用模块名表达职责，示例为 `chassis_move`、`gimbal_control`、`PowerLimit`。
- 文件内私有函数使用 `static`，文件内私有表使用 `static const`。
- 宏使用全大写和下划线，示例为 `ROBOT_CHASSIS`、`CHASSIS_CONTROL_TIME_MS`、`CHASSIS_BODY_VEL_BRAKE_ACCEL_LIMIT`。
- 循环索引按周围代码使用 `uint8_t i`；涉及数组长度和状态标志时使用明确宽度整数类型。

## 数据和状态

- 外部传入指针在函数开头做空指针保护，保护后直接 `return` 或返回零值。
- 遥控器输入先经死区处理，再进入比例换算、滤波、限幅和控制量赋值。
- 角度进入控制前使用 `rad_format` 归一化，输出速度、电流和功率进入执行前使用 `fp32_constrain`、`chassis_limit_abs` 或最大轮速比例缩放。
- 模式切换在 `*_mode_change_control_transit` 中处理，切换时保存目标角、相对角、规划速度、积分项或上一模式。
- 掉线和超时状态通过 `toe_is_error`、反馈时间戳、`online`、`error_code` 进入零输出、无力或保持逻辑。
- 底盘实测速度由轮速正解得到，调试和闭环补偿使用 `chassis_move.vx`、`chassis_move.vy`、`chassis_move.wz`。
- CAN 发送函数集中在 `*_send_cmd` 或 BSP CAN 封装中，控制函数只写目标值、中间控制量和最终电流命令。

## 注释规范

- 文件编码统一使用 UTF-8，新增注释主要使用中文。
- 文件头保留已有 Doxygen 风格；新增文件头包含 `@file`、`@brief`、`@note`。
- 对外函数使用 Doxygen 风格块注释，说明功能、输入参数、返回值和单位。
- 结构体字段和宏定义使用行尾中文注释，说明物理含义、单位、取值含义或硬件映射。
- 控制参数宏注释写清单位、作用链路和调参直接效果，示例为制动补偿、电流限幅、PID、滤波和速度规划参数。
- 运动学、闭环控制、前馈补偿和制动逻辑注释写清坐标系、正负号方向、触发条件、限幅位置和输出叠加位置。
- 复杂公式、坐标转换、运动学映射、滤波、限幅、掉线保护和模式切换需要贴近代码写清机制。
- 历史乱码注释在触碰同一区域时改回 UTF-8 中文；新增注释保持可读中文。

## 调试输出

- VOFA 输出固定 6 通道，统一经过 `VOFA_Send6`。
- VOFA 调试函数注释写清 6 个通道的信号名、单位和缩放关系。
- VOFA 默认服务函数只保留当前需要观察的一组信号，新增调试组必须保持 6 个浮点通道。
- 调试变量命名体现信号来源、目标值、反馈值和缩放倍数，示例为 `vx_plan`、`given_current_a`、`PowerLimit.P_bus`。
- 长期保留的调试接口放入 `User/Devices/vofa.*`，临时调试代码用测试宏保护后再提交。

## 代码组织

- 结构体、枚举和宏定义统一放入对应模块头文件；`.c` 文件只保留静态变量、静态表、函数声明和函数实现。
- `.c` 文件顺序为：模块头文件、依赖头文件、条件编译宏、局部宏、静态表、全局状态、静态辅助函数、任务接口实现、控制辅助函数。
- `.h` 文件顺序为：头文件保护、依赖头文件、条件编译、宏定义、枚举、结构体、外部变量声明、函数声明。
- 条件编译以 `ROBOT_TYPE`、`ROBOT_CHASSIS`、`ROBOT_GIMBAL`、`ROBOT_FRICTION` 为入口，具体实现放在对应模块文件中。
- 控制周期使用已有宏表达，示例为 `CHASSIS_CONTROL_TIME_MS`、`GIMBAL_CONTROL_TIME`、`CHASSIS_TASK_INIT_TIME`、`GIMBAL_TASK_INIT_TIME`。
- 现有 PID、滤波、遥控器、CAN、IMU、检测、功率控制和 VOFA helper 优先复用已有接口。

## 实现约束

- 新增底盘逻辑沿 `chassis_behaviour_*` -> `chassis_set_contorl` -> `chassis_control_loop` -> `chassis_send_cmd` 链路接入。
- 新增底盘运动学逻辑放入 `chassis_calculate.*`，新增底盘执行层逻辑放入 `Omni_chassis.*`。
- 新增云台逻辑沿 `gimbal_behaviour_*` -> `gimbal_set_control` -> `gimbal_control_loop` -> `gimbal_send_cmd` 链路接入。
- 新增发射逻辑沿 `shoot_init`、`shoot_control_loop`、`shoot_3508.*` 链路接入。
- 新增功率限制或电流缩放逻辑放入 `User/APP_Support/power_control`。
- CubeMX 管理的 FreeRTOS、外设、GPIO、时钟、中断、DMA、NVIC、HAL tick、USB、FDCAN、UART、SPI、TIM 参数通过 CubeMX 修改流程进入工程。
