整车核心功能框架迁移提示词：

你现在面对的是一个 RoboMaster STM32H7 FreeRTOS 工程。请先阅读新工程的 `User/APP`、`User/app_suppot'、`User/Bsp`、`User/Algorithm`、`Core/Src/freertos.c`、`Core/Src/main.c`，再按下面的框架修正新工程。实现路径必须保持模块职责一致：任务入口文件只组织固定周期控制流程，具体机构文件覆盖弱函数，行为文件只决定模式和目标量，算法和发送逻辑留在对应机构模块，参数放到该模块现有配置头文件或 `robot_param.h` 的同类位置。

核心控制范式 = 固定周期任务入口 -> 初始化 -> 模式判定 -> 模式切换过渡 -> 反馈更新 -> 目标量生成 -> 控制计算 -> 执行发送。云台入口参考 `gimbal_task()`：延时初始化后调用 `gimbal_init()` 与 `shoot_init()`，循环顺序固定为 `gimbal_set_mode()`、`gimbal_mode_change_control_transit()`、`gimbal_feedback_update()`、`gimbal_set_control()`、`gimbal_control_loop()`、`shoot_control_loop()`、`gimbal_send_cmd()`，周期由 `GIMBAL_CONTROL_TIME` 控制。底盘入口参考 `chassis_task()`：延时后 `chassis_init()`，等待电机与 DBUS 在线，循环顺序固定为 `chassis_set_mode()`、`chassis_mode_change_control_transit()`、`chassis_feedback_update()`、`chassis_set_contorl()`、`chassis_control_loop()`、`chassis_send_cmd()`，周期由 `CHASSIS_CONTROL_TIME_MS` 控制。射击入口由云台任务内调用，`shoot_control_loop()` 的顺序为 `shoot_set_mode()`、`shoot_feedback_update()`、弹仓/摩擦轮/拨弹盘状态处理、PID 计算、云台停止联锁、摩擦轮 CAN 发送、返回拨弹盘电流。

公共任务文件使用 `__weak` 声明可覆盖接口，具体机构文件使用 `__attribute__((used))` 实现同名函数。云台具体文件包括 `yaw_pitch_direct.c`、`double_yaw_pitch.c`、`Multi-axis_robotic_arm.c`，底盘具体文件包括 `Balance_chassis.c`、`Mecanum_chassis.c`、`Omni_chassis.c`、`steering_chassis.c`，射击具体文件包括 `shoot_3508.c`、`shoot_double.c`、`shoot_none.c`。新工程不需要这种宏选择，新工程只是一个兵种的工程，只包含一个云台，一个底盘，一个射击。

数据结构范式 = 每个模块有一个全局控制对象，初始化阶段绑定外部数据指针，循环阶段只更新对象字段。云台全局对象是 `gimbal_control_t gimbal_control`，字段包含遥控器指针、INS 角度指针、陀螺仪指针、yaw/pitch 电机结构、校准结构；底盘全局对象是 `chassis_move_t chassis_move`，字段包含遥控器指针、云台电机指针、INS 指针、底盘模式、滤波器、速度/角度设定、PID、对应底盘机构电机指针；射击全局对象是 `shoot_control_t shoot_control`，字段包含射击模式、遥控器指针、拨弹盘与摩擦轮电机指针、PID、按键状态、热量和堵转计数。

行为层范式 = 行为枚举 -> 行为判定函数 -> 控制模式选择 -> 行为控制函数输出目标量。云台行为在 `gimbal_behaviour.c/h`，`gimbal_behaviour_mode_set()` 先调用 `gimbal_behavour_set()` 得到 `gimbal_behaviour`，再把行为映射到 `GIMBAL_MOTOR_RAW`、`GIMBAL_MOTOR_GYRO`、`GIMBAL_MOTOR_ENCONDE`；`gimbal_behaviour_control_set()` 根据行为调用 `gimbal_zero_force_control()`、`gimbal_init_control()`、`gimbal_cali_control()`、`gimbal_relative_angle_control()`、`gimbal_spin_control()` 等函数输出 yaw/pitch 增量。底盘行为在 `chassis_behaviour.c/h`，`chassis_behaviour_mode_set()` 得到 `chassis_behaviour_mode` 后映射到 `CHASSIS_VECTOR_FOLLOW_GIMBAL_YAW`、`CHASSIS_VECTOR_FOLLOW_CHASSIS_YAW`、`CHASSIS_VECTOR_NO_FOLLOW_YAW`、`CHASSIS_VECTOR_RAW` 等底盘控制模式；`chassis_behaviour_control_set()` 输出 vx、vy、wz 或角度设定。新增行为时按“枚举值、控制函数声明、行为判定分支、控制模式映射、目标量输出分支”的顺序加入。

机构层范式 = 反馈统一归一化，目标量统一限幅，控制计算统一写入 `given_current` 或 MIT 电机 set 字段，发送函数统一做掉线和安全判断。云台反馈用编码器转换函数 `motor_ecd_to_angle_change()`、INS 角度、陀螺仪数据更新 yaw/pitch 的 `absolute_angle`、`relative_angle`、`motor_gyro`；目标量通过 `gimbal_absolute_angle_limit()` 或 `gimbal_relative_angle_limit()` 累加并 `rad_format()`；控制环按 `GIMBAL_MOTOR_RAW/GYRO/ENCONDE` 选择原始电流、绝对角度外环+速度内环、相对角度外环+速度内环；发送由 `gimbal_send_cmd()` 汇总 yaw、pitch、shoot 电流并调用 CAN。底盘反馈根据底盘类型更新电机速度、IMU 姿态、云台相对角、底盘速度估计或 VMC 状态；目标量由行为层给出后在底盘文件内完成坐标变换、滤波、限幅、平衡/VMC/LQR 或轮速解算；发送函数根据电机类型调用对应 CAN/MIT 电机发送接口。射击反馈更新拨弹盘速度/角度、摩擦轮速度、按键长按计数、热量和堵转计数；控制环用状态机控制 `SHOOT_STOP`、`SHOOT_READY_FRIC`、`SHOOT_READY_BULLET`、`SHOOT_BULLET`、`SHOOT_CONTINUE_BULLET`。

跨模块依赖范式 = 使用 getter 返回只读指针或模块对象指针，避免在无关文件复制状态。遥控器通过 `get_remote_control_point()` 获取；云台 yaw/pitch 通过 `get_yaw_motor_point()`、`get_pitch_motor_point()` 给底盘使用；INS 通过 `get_INS_angle_point()`、`get_gyro_data_point()` 给云台和底盘使用；CAN 电机反馈通过 `get_*_motor_measure_point()` 绑定；裁判功率和热量通过 `get_chassis_power_and_buffer()`、`get_shoot_heat0_limit_and_heat0()`、`get_shoot_heat1_limit_and_heat1()` 获取。新增跨模块数据时优先增加 getter 或消息结构，调用方向沿现有链路放置。

修改约束 = CubeMX 管理的外设、GPIO、时钟、DMA、NVIC、FreeRTOS 任务和 HAL tick 参数需要先给出 CubeMX 路径、建议值、生成影响文件，再等待确认；业务逻辑修改放在 `User` 下对应模块；Vofa 调试输出最多 6 通道；新增注释只解释复杂控制意图；新增参数放到现有配置头的同类区域；新增函数、宏、结构体字段使用周围命名、缩进、类型和调用顺序。

裁判系统接收处理链路核对提示词：

你现在需要核对新工程的 RoboMaster 裁判系统链路是否与参考工程一致。请读取 `User/APP/referee_usart_task.c/h`、`User/APP/referee.c/h`、`User/APP/protocol.h`、`User/Bsp/bsp_usart.c/h`、`User/Controller/fifo.c/h`、`Core/Src/main.c`、`Core/Src/freertos.c`、`Core/Src/usart.c`，并按下面的链路逐项核对。

协议定义在 `protocol.h`：帧头 SOF 为 `0xA5`，帧头长度 `REF_PROTOCOL_HEADER_SIZE = 5`，cmd_id 长度 `REF_PROTOCOL_CMDID_SIZE = 2`，CRC16 长度 `REF_PROTOCOL_CRC16_SIZE = 2`，最大帧长 `REF_PROTOCOL_FRAME_MAX_SIZE = 192`，帧结构为 SOF、data_length、seq、CRC8、cmd_id、data、CRC16；`referee_cmd_id_e` 枚举保存 `0x0001` 到 `0x0309` 的命令 ID；所有数据结构使用 `#pragma pack(push, 1)` 保证字节对齐。

接收缓存和 FIFO 在 `bsp_usart.c/h`：`usart1_buf[2][USART_RX_BUF_LENGHT]` 是串口接收双缓冲，`referee_fifo_buf[REFEREE_FIFO_BUF_LENGTH]` 是 FIFO 存储区，`fifo_s_t referee_fifo` 是解包输入队列。任务初始化在 `referee_usart_task()` 与 `main.c` 都会调用 `fifo_s_init(&referee_fifo, referee_fifo_buf, REFEREE_FIFO_BUF_LENGTH)`；当前 `main.c` 中 USART1 空闲 DMA 接收调用为 `HAL_UARTEx_ReceiveToIdle_DMA(&huart1, gimbal_receive_data, CHASSIS_DATA_LENGTH)`，`bsp_usart.c` 的 USART1 空闲回调当前调用 `gimbal_memcpy_data_from_chassis()`，裁判入队代码 `HAL_UARTEx_ReceiveToIdle_DMA(&huart1, usart1_buf[x], USART_RX_BUF_LENGHT)` 与 `fifo_s_puts(&referee_fifo, ...)` 处于注释状态。迁移核对时需要确认新工程用于裁判的串口是否真正把接收字节写入 `referee_fifo`，接收重启是否发生在空闲回调和错误回调内。

解包任务在 `referee_usart_task.c`：`referee_usart_task()` 先调用 `init_referee_data()` 清零裁判数据，再初始化 FIFO，然后每 10ms 调用 `referee_unpack_fifo_data()`。解包状态机使用 `unpack_data_t referee_unpack_data`，从 `referee_fifo` 单字节取数，按 `STEP_HEADER_SOF`、`STEP_LENGTH_LOW`、`STEP_LENGTH_HIGH`、`STEP_FRAME_SEQ`、`STEP_HEADER_CRC8`、`STEP_DATA_CRC16` 推进；收到 `0xA5` 后缓存帧头，长度低高字节合成 `data_length`，长度小于 `REF_PROTOCOL_FRAME_MAX_SIZE - REF_HEADER_CRC_CMDID_LEN` 后继续，帧头长度达到 5 字节时调用 `verify_CRC8_check_sum()`，整帧达到 `REF_HEADER_CRC_CMDID_LEN + data_length` 时调用 `verify_CRC16_check_sum()`，CRC16 通过后调用 `referee_handle_data(p_unpack->protocol_packet)`。

数据分发在 `referee.c`：`init_referee_data()` 清零 `referee_receive_header`、`referee_send_header`、`game_status`、`game_result`、`game_robot_HP`、`event_data`、`referee_warning`、`dart_info`、`robot_status`、`power_heat_data`、`robot_pos`、`buff`、`hurt_data`、`shoot_data`、`projectile_allowance`、`rfid_status`、`dart_client_cmd`、`ground_robot_position`、`radar_mark_data`、`sentry_info`、`radar_info`、`robot_interaction_data`、`custom_robot_data`、`map_command`、`remote_control`、`map_robot_data`、`custom_client_data`、`map_data_`、`custom_info`、`robot_custom_data`，并把 `g_referee_last_rx_ms` 置 0。`referee_handle_data()` 先记录 `g_referee_last_rx_ms = HAL_GetTick()`，复制 5 字节帧头，读取 2 字节 `cmd_id`，再用 `switch(cmd_id)` 把 payload 复制到对应全局结构体。

业务读取接口在 `referee.c/h`：`get_chassis_power_and_buffer()` 输出 `robot_status.chassis_power_limit` 和 `power_heat_data.buffer_energy`；`get_robot_id()` 输出 `robot_status.robot_id`；`referee_data_available(timeout_ms)` 先校验 robot_id 是否在红方 1-7 或蓝方 101-107，再按 `g_referee_last_rx_ms` 和超时时间判断可用性；`get_shoot_heat0_limit_and_heat0()` 输出 `robot_status.shooter_barrel_heat_limit` 与 `power_heat_data.shooter_17mm_1_barrel_heat`；`get_shoot_heat1_limit_and_heat1()` 输出 `robot_status.shooter_barrel_heat_limit` 与 `power_heat_data.shooter_42mm_barrel_heat`。
